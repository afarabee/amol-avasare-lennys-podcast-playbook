# Product-Minded Engineer Delegation

## Purpose
Use this skill to decide when engineers can drive work end to end and when PM ownership is still required.

## Use this skill when
- PM bandwidth is constrained
- Engineers are highly product-minded
- Small efforts are slowed by overreliance on PMs
- The team wants a clean ownership rule

## Do not use this skill when
- Engineers are not equipped to handle stakeholder alignment
- The work is highly controversial, externally visible, or high-risk
- Regulatory or cross-functional complexity is substantial

## Core principles
1. Small work does not always need dedicated PM execution.
2. Product-minded engineers can own small efforts.
3. PMs should concentrate on higher-leverage ambiguity and alignment work.
4. Size thresholds are heuristics, not laws.
5. Controversy and risk override duration.

## Workflow
1. Estimate effort size.
2. Assess risk, controversy, and stakeholder surface area.
3. Decide ownership:
   - engineer-led
   - PM-advised
   - PM-led
4. Clarify what the engineer owns:
   - scope
   - stakeholder coordination
   - security/legal touchpoints
   - launch follow-through
5. Define escalation triggers for PM involvement.

## Output format
Return:
- initiative size
- recommended owner
- why
- required stakeholders
- escalation triggers
- support model

## Anti-patterns
- Delegating work purely by size while ignoring sensitivity
- Assuming all engineers are naturally product-minded
- Pulling PMs out too early from important work
- Using this model without explicit expectations
