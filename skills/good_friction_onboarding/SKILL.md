# Good Friction Onboarding

## Purpose
This skill helps design or evaluate onboarding flows using the principle that the best onboarding is not always the shortest onboarding. Use friction intentionally when it improves user understanding, product-user fit, or conversion quality.

## Use this skill when
- A user asks to improve onboarding, signup, activation, or first-run experience
- A team wants to reduce drop-off and is assuming the answer is fewer steps
- There is a need to personalize onboarding based on user type, job-to-be-done, or intent
- A product has broad capabilities and users need routing to the most relevant path

## Do not use this skill when
- The product has a single, obvious use case with no routing need
- The onboarding friction is clearly redundant, repetitive, or compliance-only with no user value
- The request is only about visual polish and not flow design

## Core principles
1. Remove friction that adds no value.
2. Keep or add friction that improves relevance, confidence, or routing.
3. Break high cognitive-load steps into smaller chunks.
4. Capture only the information that will be used immediately or downstream.
5. Optimize for activation quality, not just step completion.

## Workflow
1. Identify the user’s likely intent at entry.
2. Determine whether the product has multiple paths, audiences, or use cases that require routing.
3. Audit each onboarding step:
   - What does this step teach?
   - What does this step learn?
   - What downstream action becomes better because of this step?
4. Classify each step:
   - Remove
   - Keep
   - Split into smaller steps
   - Rewrite for clarity
   - Add a new step for segmentation/routing
5. Design the output path:
   - Recommended feature
   - Recommended workflow
   - Recommended content/example
6. Propose an experiment plan:
   - hypothesis
   - expected activation effect
   - risks
   - metrics

## Output format
Return:
- current issue
- recommended friction changes
- rationale for each step
- experiment design
- success metrics
- anti-patterns to avoid

## Anti-patterns
- “Fewer clicks is always better”
- Asking segmentation questions and never using the answers
- Long forms with no obvious payoff
- Forcing all users through the same first-run path
- Optimizing only for signup completion while hurting activation
