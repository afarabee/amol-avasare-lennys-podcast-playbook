# Proportionate Process

## Purpose
This skill helps decide how much process a given initiative actually needs, so teams move quickly without creating preventable confusion.

## Use this skill when
- A team is unsure whether a project needs a PRD, kickoff, or just Slack coordination
- Small projects are getting bogged down by heavyweight process
- Leaders want a clean decision rule for PM vs engineer-led work
- A request asks how to scale execution without bureaucracy

## Do not use this skill when
- The work is highly regulated and documentation is mandatory
- The project has major external dependency or executive visibility and clearly needs formal handling
- The request is specifically for a full PRD

## Core principles
1. Not every initiative needs a PRD.
2. Small work should move through lightweight mechanisms.
3. Large or risky work deserves an early cross-functional kickoff.
4. Engineers can lead small efforts if they are product-minded.
5. PM time should go to ambiguity, leverage, and coordination where it matters most.

## Workflow
1. Assess the initiative on:
   - size
   - risk
   - controversy
   - cross-functional surface area
   - reversibility
2. Classify it:
   - lightweight
   - moderate
   - heavyweight
3. Recommend the minimum viable process:
   - lightweight: Slack/thread/prototype
   - moderate: brief scoped doc + async review
   - heavyweight: kickoff + documented rationale + stakeholder alignment
4. Clarify ownership:
   - engineer-led
   - PM-led
   - shared
5. Return the recommended execution plan.

## Output format
Return:
- initiative type
- process level recommended
- why that level fits
- required stakeholders
- suggested artifacts
- ownership model
- risks if under-processed or over-processed

## Anti-patterns
- Requiring PRDs for everything
- Treating Slack-only coordination as sufficient for high-risk work
- Confusing bureaucracy with rigor
- Pulling PMs deeply into every tiny change
